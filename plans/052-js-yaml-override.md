## Why this matters

`pnpm audit` reports a HIGH advisory (GHSA-52cp-r559-cp3m, quadratic CPU on
YAML merge keys) for `js-yaml` <4.3.0, reached through
`@tanstack/react-start > @tanstack/start-plugin-core > xmlbuilder2 > js-yaml`.

**It is not reachable in this app**, and the plan is worth doing anyway. The
only call site is TanStack Start's sitemap generation, which runs at build time
and is gated behind `startConfig.sitemap?.enabled` — and `vite.config.ts` calls
`tanstackStart()` with no options at all, so that branch never executes.
Nothing in a served request handler touches it.

The cost of leaving it is that `pnpm audit` has a standing HIGH finding in it.
An audit output nobody can act on is an audit output people stop reading, and
the next genuinely-reachable advisory hides in the noise. Clearing it is one
line.

## Current state

`package.json` has **no** `pnpm` section and no `overrides` key today —
verified by grep. `packageManager` is at line 9:

```json
  "packageManager": "pnpm@10.34.5",
```

`pnpm audit` at plan time reports **3 vulnerabilities: 1 low, 1 moderate,
1 high**. The other two are `esbuild` advisories reached only through
`drizzle-kit`, which is a `devDependency` used for `db:generate`/`db:migrate`
and never bundled into the production server. Those two are deliberately **not**
addressed here — they were re-verified as dev-only during the audit that
produced this plan.

`xmlbuilder2@4.0.3` itself declares `"js-yaml": "^4.1.1"`, so an override to
`>=4.3.0` stays inside the range the library already accepts. No API used here
changes between 4.2 and 4.3.

### Constraints from `CLAUDE.md` you must honor

Read `CLAUDE.md` before starting. Two rules bear directly on this plan:

- **pnpm stays on 10.x.** Do not "upgrade" `packageManager` to 11.x. Every
  11.x release breaks the Railway deploy — Nixpacks' corepack loads
  `bin/pnpm.cjs` without a dynamic-import callback, and 11.x opens with
  `import('./pnpm.mjs')`. CI will not catch this because GitHub Actions
  installs pnpm a different way. This plan touches `package.json`; leave that
  field exactly as it is.
- **`nitro` is aliased to an exact `nitro-nightly` build on purpose.** Do not
  let any install re-resolve it. After installing, verify it is unchanged
  (Step 3).

## Steps

### Step 1: Record the baseline

Run `pnpm audit` and note the exact summary line.

**Verify**: output ends with a line reporting 3 vulnerabilities including
1 high. If the counts differ from "1 low | 1 moderate | 1 high", the
dependency tree has moved since this plan was written — see STOP conditions.

### Step 2: Add the override

Add a top-level `pnpm` section to `package.json`. Place it after
`packageManager` so related tooling config stays together:

```json
  "pnpm": {
    "overrides": {
      "js-yaml": ">=4.3.0"
    }
  },
```

Then run `pnpm install`.

**Verify**: `pnpm install` exits 0 and `pnpm-lock.yaml` is modified
(`git status --short` shows it).

### Step 3: Confirm nitro did not move

`CLAUDE.md` warns that any `pnpm` operation can silently re-resolve the
`nitro-nightly` pin, which would change the server runtime.

**Verify**: `git diff pnpm-lock.yaml | grep -i "nitro"` → **no output**.
If nitro appears in the diff, STOP.

### Step 4: Confirm the advisory is gone

**Verify**: `pnpm audit` → the high-severity `js-yaml` advisory
(GHSA-52cp-r559-cp3m) is no longer listed. The two `esbuild` advisories
should still be present and are expected.

### Step 5: Full gate

**Verify**, in order, all exit 0:

```
pnpm install --frozen-lockfile
pnpm generate-routes
pnpm typecheck
pnpm lint
pnpm test
pnpm build
```

`pnpm install --frozen-lockfile` passing is the important one — it proves the
committed lockfile matches `package.json`, which is what CI runs.

## STOP conditions

Stop and report back (do not improvise) if:

- `pnpm audit` at Step 1 does not show a high-severity `js-yaml` advisory.
  Upstream may have fixed it already, in which case this plan is unnecessary
  and the override would be dead config.
- The nitro check at Step 3 shows nitro moved in the lockfile.
- `pnpm build` fails after the override. That would mean the sitemap/xmlbuilder
  path is reachable in a way this plan's analysis missed — report the error
  rather than pinning js-yaml back or adding more overrides.
- You find yourself wanting to add a second override to silence the esbuild
  advisories. That is explicitly out of scope.

## Maintenance notes

For whoever owns this next:

- **The override is a stopgap, not a fix.** When `@tanstack/react-start` ships
  a release whose transitive `xmlbuilder2` already pulls js-yaml >=4.3.0, this
  entry becomes dead weight and should be removed. Check on the next TanStack
  bump.
- **If sitemap generation is ever enabled** (`tanstackStart({ sitemap: { enabled: true } })`
  in `vite.config.ts`), this dependency stops being unreachable and the
  override stops being merely cosmetic — it becomes load-bearing. Note that in
  the same commit that enables it.
- A reviewer should check one thing: that the lockfile diff contains no `nitro`
  lines. That pin is the single most fragile thing in this repo's dependency
  graph, and `CLAUDE.md` explains why at length.
