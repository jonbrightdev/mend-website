## Why this matters

This repo (the Mend website + portal: TanStack Start, Better Auth, Drizzle,
Postgres/PGlite) has **no tests and no CI** — the only verification gate is
`pnpm typecheck`. Several upcoming fixes (transactional ingest, ingest input
limits, dashboard query rewrite — plans 010, 011, 015) change data-handling
logic that typecheck cannot protect. This plan adds Vitest, extracts the ingest
parsing logic into a testable module, adds unit tests plus a real-database test
harness (in-memory PGlite), and adds a GitHub Actions workflow running
typecheck + tests + build on every push/PR.

### Step 3: Unit tests for the pure modules

Create the three unit-test files. Cover at minimum:

- `src/lib/ingest-payload.test.ts`
  - `parsePayload` happy path: a full valid payload (url, startedAt epoch-ms,
    pageTitle, issues with all fields) returns the parsed shape; `scannedAt`
    is a `Date` equal to `new Date(startedAt)`.
  - Rejections (each throws `IngestError` with a useful message): non-object
    body; missing/non-string `url`; non-http(s) url (`ftp://x`); missing

    empty `ruleId`.
  - Defaults: missing `pageTitle` falls back to the url; missing `domOrder`
    falls back to array index; non-string entries in `wcag` are filtered out.
  - `groupViolations`: issues with the same `ruleId` group into one violation;
    nodes are sorted by `domOrder`; `tags` = category of first issue plus
    de-duplicated wcag entries; distinct ruleIds produce distinct violations.
- `src/lib/dashboard-data.test.ts`
  - `byRule` aggregates node counts and page counts across audits and sorts by
    impact rank then count.
  - `aggregateTrend` sums `history[i]` across audits per run date.
  - `wcagUnderstandingUrl("1.1.1 Non-text Content (A)")` →
    `https://www.w3.org/WAI/WCAG21/Understanding/non-text-content.html`;
    unknown criterion → `null`.
  - `ruleSpecFor` returns the catalogue entry for a known ruleId and a
    synthesized spec (with the fallback `fix` text) for an unknown one.
- `src/lib/api-key.test.ts`
  - `generateKey()` matches `/^mend_[A-Za-z0-9_-]{43}$/` and two calls differ.
  - `hashKey("mend_test")` returns a 64-char lowercase hex string and is
    deterministic (same input → same output).

**Verify**: `pnpm test` → all tests pass.

### Step 4: PGlite test-database harness

Create `src/test/db.ts`:

```ts
// In-memory Postgres for tests. Call createTestDb() BEFORE importing any
// module that imports "@/db": src/db/index.ts binds `db` to
// globalThis.__mendDb ??= ..., so pre-setting the global makes the entire
// app use this instance. Schema comes from replaying drizzle/*.sql in order.
import { readdirSync, readFileSync } from "node:fs";
import { join } from "node:path";
import { drizzle } from "drizzle-orm/pglite";
import { PGlite } from "@electric-sql/pglite";
import * as schema from "@/db/schema";

export async function createTestDb() {
  const client = new PGlite(); // no data dir → in-memory
  const dir = join(process.cwd(), "drizzle");
  const migrations = readdirSync(dir).filter((f) => f.endsWith(".sql")).sort();
  for (const file of migrations) {
    // drizzle-kit separates statements with this marker in generated SQL
    const sql = readFileSync(join(dir, file), "utf8");
    for (const stmt of sql.split("--> statement-breakpoint")) {
      if (stmt.trim()) await client.exec(stmt);
    }
  }
  const db = drizzle(client, { schema });
  (globalThis as Record<string, unknown>).__mendDb = db;
  return db;
}
```

(Adjust the statement-splitting if the generated SQL in `drizzle/*.sql` uses a
different breakpoint marker — open `drizzle/0000_mushy_hellfire_club.sql` and
check; drizzle-kit's default is `--> statement-breakpoint`.)

Create `src/test/db.test.ts`: a smoke test that `createTestDb()` resolves, then
inserts a row into `user` and one into `audit` (import the tables from
`@/db/schema`) and selects them back. Use dynamic `await import("@/db")` *after*
calling `createTestDb()` if you need the singleton, or use the returned `db`
directly.

**Verify**: `pnpm test` → all tests pass, including the db smoke test.
**Verify**: `ls .data 2>/dev/null` inside the repo shows no *new* files created by the test run (in-memory PGlite must not write to disk; a pre-existing `.data` dir from dev use is fine).

### Step 5: GitHub Actions workflow

Create `.github/workflows/ci.yml`:

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck
      - run: pnpm test
      - run: pnpm build
```

Note: `pnpm-workspace.yaml` already has `allowBuilds` entries for `esbuild`,
`sharp`, and `@electric-sql/pglite`, so `pnpm install` in CI can run their
build scripts without prompting. Do not add new entries.

**Verify**: `pnpm install --frozen-lockfile && pnpm typecheck && pnpm test && pnpm build` locally → all exit 0 (this is exactly what CI will run).
